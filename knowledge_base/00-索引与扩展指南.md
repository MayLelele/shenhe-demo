# 索引与扩展指南

## 1. 本知识库的定位

- **给人读**：统一口径、培训、与业务对需求。
- **对代码的映射**：`lookup_policy_rule` 的命中依赖 `tools.py` 中 `POLICY_RULES` 的**条目标题与关键词**；知识库 `01-` 文档是「完整表述」，代码里宜保持**可检索的短摘要**（勿混进未上线的条款）。

## 2. 如何增加一条新审核规则

1. 复制 [`entries/_TEMPLATE_规则条目.md`](./entries/_TEMPLATE_规则条目.md) 到同目录，命名为 `YYYYMMDD-短标题.md` 或 `场景名.md`。
2. 按模板补全：触发条件、需核对事实、可能结论、与系统/备注的依赖、例外情况。
3. 在 [`01-提币KYC-规则条目.md`](./01-提币KYC-规则条目.md) 增加一节索引（链接到你的新文件，或把成熟正文合并进 01-）。
4. 若需 Agent/审核台在检索中命中，在 `audit_agent/tools.py` 的 `POLICY_RULES` 增加键值；键名应便于关键词匹配（见工具 `lookup_policy_rule` 逻辑）。
5. 变更记录可写在当次 PR 或 [`../执行纪要-复盘.md`](../执行纪要-复盘.md) 同类文档中，便于审计追溯。

## 3. 若「审核结果」需沉淀为知识

- 对**可复用**的判定模式：抽象成规则条目，避免复制个案流水。
- 对**仅个案**的结论：可记入 `entries/cases/`（可自行建目录）并打标签，不强行进 `POLICY_RULES`。
- 项目内 [`../SKILL.md`](../SKILL.md) 约定：人工校准后把结构化内容写入本知识库，与上述流程一致。

## 4. 版本与生效范围

| 项 | 说明 |
|----|------|
| 仓库内 | 以 Git 提交为版本；大版本可在 `01-` 文首写变更列表 |
| 对外生效 | 以公司制度与线上规则引擎为准；本库为**协作草稿与对齐用** |
